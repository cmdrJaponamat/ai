## Язык И Стиль

- По умолчанию всегда работать на русском языке.
- На другой язык переходить только если пользователь явно попросил.
- Не смешивать русский и английский без необходимости.
- Отвечать кратко, практично и по делу.

## Главные Источники Контекста

- `~/ai` является главным operational control plane.
- Основной workflow-файл: `~/ai/WORKFLOW.md`.
- Реестр проектов: `~/ai/PROJECT_CONTEXTS.md`.
- Индекс проектов: `~/ai/projects.db`.
- Хронологический журнал действий: `~/ai/actions.log`.
- Тематические recovery-файлы: `~/ai/recovery/`.

## Правила Загрузки Контекста

- Нельзя загружать все project recovery/context файлы подряд по умолчанию.
- Сначала нужно смотреть индекс проектов в `~/ai/PROJECT_CONTEXTS.md`.
- Проектный контекст нужно открывать только когда он реально нужен текущей задаче.
- Если неясно, какой проект относится к задаче, нужно задать короткий уточняющий вопрос.
- Всегда выбирать минимально достаточный контекст.

## Что Считать Проектом

- Проектом считается только каталог, в котором есть `.git`.
- У проекта должен быть remote, привязанный к `cmdrJaponamat`.
- База проектов служит только индексом и навигацией.
- Основной проектный recovery-контекст хранится в Markdown-файле внутри самого проекта.

## Правила Для Проектов

- При создании нового проекта сразу создавать `project-root/.ai-recovery.md`.
- Сразу регистрировать этот recovery-файл в `~/ai/PROJECT_CONTEXTS.md`.
- В recovery-файле проекта обязательно должны быть:
  - путь к проекту
  - назначение проекта
  - текущее состояние
  - важные команды
  - следующие шаги

## Git-Политика

- Если работа идет внутри git-репозитория, приоритетом являются дискретные коммиты завершенных изменений.
- Перед коммитом нужно проверять изменения.
- Нельзя смешивать в одном коммите несвязанные правки.
- Если git-репозитория нет, нужно спросить пользователя, создавать ли его.
- Нельзя молча делать `git init`.

## Модель Работы С Системой

- Любая работа вне обычного проектного git-репозитория считается system work.
- `~/ai` является управляющим слоем для system work.
- Любое системное действие должно оставлять след в `~/ai/actions.log`.
- Любое нетривиальное системное изменение должно иметь recovery trail в `~/ai/recovery/` или обновление существующей recovery-заметки.

## Что Обязательно Фиксировать Для System Work

- что изменено
- зачем изменено
- как проверено
- как откатить или что откат только ручной
- нужен ли relogin или reboot

## Что Считать System Work

- изменения в `~/.config`
- user services в systemd
- shell-конфиги
- установка и удаление пакетов
- аудио, видео, ввод
- сетевые настройки
- autostart
- WM, compositor, desktop environment
- скрипты в `~/.local/bin`
- bootstrap и миграция окружения

## Политика Отката

- Если изменение git-managed, откат предпочтительно делать через git.
- Если изменение вне git, нужно оставить rollback-команду, чеклист или явную пометку, что откат ручной.
- Нельзя оставлять системное изменение без понятного recovery trail.

## Безопасность

- Нельзя выполнять разрушительные операции без явного подтверждения пользователя.
- Нельзя без причины массово сканировать весь домашний каталог.
- Нужно предпочитать зарегистрированные recovery-файлы и индексы, а не догадки.

## Идеи Для Пет-Проектов

- Любую стоящую идею нужно фиксировать в `~/pet`.
- Для идеи нужно кратко указывать:
  - цель
  - почему она интересна
  - возможный стек
  - минимальный MVP

## Планы На Перспективу

- Тему миграции окружения на другой ПК нужно держать в планах.
- `~/dotfiles` считается ключевым проектом для этой миграции.
- Зависимость от железа нужно максимально уменьшать.
- Все нужно постепенно приводить к путям, завязанным на `$HOME`, а не на кривые абсолютные legacy-пути.
