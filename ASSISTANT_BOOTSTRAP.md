# Базовый Контекст Ассистента

Дата создания: 2026-03-20
Владелец: japonamat
Область: постоянный стартовый контекст для терминальных LLM-сессий

## Язык И Стиль

- Всегда работай строго на русском языке, если пользователь явно не попросил другое.
- Не смешивай русский и английский без необходимости.
- Отвечай кратко, практично и по делу.

## Роль

- Считай `~/ai/` главным источником постоянных правил, workflow, recovery-контекста и ссылок на проекты.
- Считай `~/ai/` operational control plane для работы с системой вне обычных проектных репозиториев.
- Используй `~/ai/WORKFLOW.md` как основной workflow-файл, если более узкий файл в `~/ai/` не задает правила для текущей задачи.
- Если доступен `~/ai/RULES.json`, считай его компактным машинным источником базовых правил.
- Если доступны `~/ai/system_inventory.json` и `~/ai/system_inventory_human.md`, считай их первичным источником сведений о пакетах, версиях, бинарях и базовых user services.
- Не полагайся на память прошлых запусков, если нужный контекст можно явно прочитать из `~/ai/`.

## Правила Загрузки Контекста

- Не читай все связанные project context/recovery файлы по умолчанию.
- Сначала смотри `~/ai/PROJECT_CONTEXTS.md`, чтобы понять, какие проекты и recovery-файлы существуют.
- Загружай project-specific контекст только если он реально относится к текущей задаче или если пользователь явно назвал проект.
- Если непонятно, какой проектный контекст нужен, задай короткий уточняющий вопрос вместо массовой загрузки всего подряд.
- Предпочитай минимально достаточный контекст: только те файлы, которые нужны для текущей задачи.

## Правила Для Проектов

- При создании нового проекта сразу создавай recovery/context файл в формате Markdown в корне проекта.
- Сразу добавляй ссылку на этот recovery-файл в `~/ai/PROJECT_CONTEXTS.md`.
- При создании нового проекта сразу обновляй и `~/ai/projects.db` как индексный реестр.
- Для этого предпочитай использовать локальный скрипт регистрации проекта, если он доступен.
- Для каждого проекта должен быть один явный source of truth:
  - по умолчанию это `project-root/.ai-recovery.md`
  - `ROADMAP.md`, `PRD.md` и другие документы вторичны и обязаны соответствовать recovery
- В recovery-файлах должны быть:
  - путь к проекту
  - назначение проекта
  - текущее состояние
  - важные команды
  - следующие шаги

## Git-Политика

- Если текущая задача выполняется внутри git-репозитория, приоритетом является коммитить законченные дискретные изменения.
- Если репозиторий уже существует и пользователь явно не запретил push, завершенный шаг по умолчанию должен заканчиваться commit + push.
- Перед коммитом нужно проверить, что изменилось, и не смешивать в одном коммите несвязанные правки.
- Если проект не находится под git, нужно явно спросить пользователя, не надо ли создать для него репозиторий.
- Нельзя молча инициализировать git-репозиторий без просьбы или подтверждения пользователя.

## Правила Для Кода

- При coding work вести `refactor counter`.
- Увеличивать счетчик на `1` после каждого завершенного дискретного feature/fix шага.
- Когда счетчик достигает `3`, перед продолжением делать отдельный рефакторинговый проход и затем сбрасывать счетчик в `0`.
- Не ждать переполнения счетчика, если уже видны дублирование, расползание условной логики или плохие границы ответственности.
- После рефакторинга явно фиксировать его в проектном recovery-контексте и, если есть git, отдельным коммитом или явно выделенной частью серии коммитов.

## Правила Для Работы С Системой

- Любая работа вне обычного проектного git-репозитория считается system work.
- Для system work `~/ai/` является управляющим слоем и местом обязательного recovery trail.
- Если инвентарь системы свежий, не перепроверяй повторно установку и версии пакетов без необходимости.
- Используй TTL инвентаря:
  - package_versions: 72 часа
  - user_services: 24 часа
  - tool_paths: 72 часа
- Если inventory явно invalidated или помечен stale, не доверяй ему слепо даже до истечения TTL.
- Перепроверяй точечно только если нужная секция inventory протухла, inventory invalidated, пользователь явно просит проверить, задача чувствительна к версии или есть признаки, что окружение изменилось.
- После заметных системных изменений обновляй инвентарь системы.
- Любое системное действие должно оставлять запись в `~/ai/actions.log`.
- Любое нетривиальное системное изменение должно сопровождаться recovery-заметкой в `~/ai/recovery/` или обновлением существующей.
- Для каждого системного изменения нужно фиксировать:
  - что изменено
  - зачем изменено
  - как проверено
  - как откатить или что откат только ручной
  - нужен ли relogin или reboot
- Если изменение находится в git-managed области, откат предпочтительно строить через git.
- Если изменение вне git, нужно оставить rollback-команду, чеклист или явную пометку, что откат неавтоматический.

## Безопасность И Область Работы

- Не выполняй разрушительные операции без явного подтверждения пользователя.
- Не сканируй массово весь домашний каталог ради контекста, если задача этого прямо не требует.
- Предпочитай recovery-файлы и зарегистрированные ссылки в `~/ai/`, а не неявную память или догадки.

## Реестр Проектов

- `~/ai/PROJECT_CONTEXTS.md` является реестром известных проектов и их recovery-файлов.
- В этом файле должны храниться ссылки и краткие описания, а не полный контекст всех проектов целиком.
